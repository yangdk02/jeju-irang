# 제주아이랑 발표 스크립트

## 발표 운영

- 목표 시간: 약 19분 30초
- 여유 시간: 약 30초
- 실제 시연: 슬라이드 31~33에서 웹 앱으로 전환
- 시연이 지연되면 각 시나리오의 마지막 단계만 생략하고 제출용 캡처로 설명
- 슬라이드 상단의 섹션 중제목과 진행바를 전환 지점에서만 짧게 언급하고 매번 읽지는 않는다.
- 출처는 화면에만 표시하고 발표 중에는 기관명과 핵심 의미만 짧게 언급한다.

---

## 표지

### 슬라이드 1. 제주아이랑 — 15초

안녕하세요. 아이와 함께 갈 제주 장소를 가족의 조건에 맞춰 찾을 수 있는 서비스, 제주아이랑을 소개하겠습니다. 화면의 QR 코드로 서비스를 직접 열어보실 수 있습니다.

### 슬라이드 2. 목차 — 25초

먼저 아이 동반 외출의 불편과 핵심 사용자를 살펴보겠습니다. 이어서 이를 해결하는 기능, 바이브코딩을 활용한 개발과 협업, 데이터 구조를 설명하겠습니다. 실제 사용자 시나리오 세 가지를 시연한 뒤 문제 해결 사례와 현재 한계, 다음 계획으로 마무리하겠습니다.

## 01 기획 배경

### 슬라이드 3. 외출 결정의 조건 — 20초

아이와 외출할 때는 장소 이름만 알면 끝나지 않습니다. 아이 연령에 맞는지, 비가 와도 갈 수 있는지, 유모차와 수유실은 가능한지, 주차와 할인, 예약까지 함께 확인해야 합니다.

### 슬라이드 4. 흩어진 정보 — 30초

문제는 이 정보가 한곳에 있지 않다는 점입니다. 지도에서 위치를 찾고, 블로그와 맘카페에서 후기를 확인하고, 다시 시설 홈페이지로 이동해 운영시간과 예약 여부를 봐야 합니다. 보호자는 한 장소를 결정하기 위해 같은 장소를 여러 번 검색하게 됩니다.

### 슬라이드 5. 경험에서 출발 — 25초

이 불편은 아이를 키우는 팀원이 제주에서 직접 경험했습니다. 마음에 드는 장소를 찾은 뒤에도 아이 나이에 맞는지, 유모차를 사용할 수 있는지, 도민 할인과 예약 여부를 다시 찾아야 했습니다. 이 반복 탐색을 줄이는 것이 출발점이었습니다.

### 슬라이드 6. 기존 서비스의 빈틈 — 30초

기존 플랫폼은 각각 강점이 있습니다. 당근과 맘카페에는 지역 후기가 있고, 인스타그램과 블로그에는 사진과 경험이 있으며, 지도와 관광 포털에는 위치와 기본정보가 있습니다. 제주아이랑은 이들을 대체하기보다 아이 동반 결정에 필요한 정보를 공통 기준으로 연결합니다.

### 슬라이드 7. 제주 유소년 — 25초

2024년 제주에는 0세부터 14세까지 유소년 8만 4천 명이 생활하고 있습니다. 제주 전체 인구의 12.4퍼센트입니다. 제주 안에서 아이와 갈 곳을 반복해서 찾는 생활 수요가 존재한다는 것을 보여줍니다.

### 슬라이드 8. 가족 여행 수요 — 25초

관광 수요도 가족 중심입니다. 2024년 제주 내국인 관광객의 50.5퍼센트가 가족이나 친지와 함께 방문했고, 평균 동반 인원은 4.4명이었습니다. 따라서 제주 거주자뿐 아니라 가족 여행객도 중요한 사용자입니다.

### 슬라이드 9. 온라인 탐색 — 30초

제주 내국인 관광객의 75.1퍼센트는 인터넷 사이트와 앱에서 정보를 찾았습니다. 온라인 정보 수집자 중 78.2퍼센트는 포털 검색을 이용했습니다. 필요한 정보가 통합되어 있기보다 사용자가 여러 검색 결과를 직접 조합하고 있다는 뜻입니다.

### 슬라이드 10. 방문지 정보 수요 — 25초

여행객이 가장 필요하다고 답한 정보는 방문지 정보였습니다. 이 비중은 2022년 34.4퍼센트에서 2024년 50.7퍼센트로 16.3퍼센트포인트 증가했습니다. 장소를 실제로 결정하는 정보의 중요성이 커진 것입니다.

### 슬라이드 11. 핵심 사용자 — 35초

통계를 바탕으로 두 사용자 유형을 정했습니다. 첫 번째는 주말이나 방학마다 아이와 갈 곳을 찾는 제주 거주 보호자입니다. 두 번째는 제한된 일정 안에서 장소와 이동 동선을 결정해야 하는 아이 동반 제주 여행객입니다. 두 사용자 모두 빠른 조건 검색과 정확한 방문정보가 필요합니다.

## 02 핵심 기능

### 슬라이드 12. 해결 흐름 — 25초

제주아이랑의 해결 방식은 네 단계입니다. 가족의 조건으로 검색하고, 장소를 같은 기준으로 비교하고, 마음에 든 곳을 저장하고, 공식 홈페이지와 예약 페이지까지 바로 연결합니다.

### 슬라이드 13. 조건 검색 — 25초

사용자는 지역, 실내외, 시설유형을 선택할 수 있습니다. 여기에 수유실, 유모차 대여, 기저귀 교환대, 주차, 입장료와 도민 할인 같은 아이 동반 조건을 더해 필요한 장소만 찾을 수 있습니다.

### 슬라이드 14. 세 가지 보기 — 25초

검색 결과는 갤러리, 표, 지도 중 편한 방식으로 볼 수 있습니다. 위치 사용을 허용하면 현재 위치에서 가까운 순서로 정렬할 수 있습니다. 현재 위치는 거리 계산에만 사용하고 별도로 저장하지 않습니다.

### 슬라이드 15. 장소 상세 — 25초

장소 상세 화면에서는 운영시간, 요금, 주차, 연령제한과 가족 편의시설을 함께 확인합니다. 더 정확한 확인이 필요하면 공식 홈페이지나 예약 페이지로 바로 이동할 수 있습니다.

### 슬라이드 16. 즐겨찾기 — 25초

마음에 든 장소는 별도 회원가입 없이 저장할 수 있습니다. 사용자는 자신만의 카테고리와 메모로 다음 나들이 후보를 정리하고, 필요하면 즐겨찾기 목록을 CSV로 내려받을 수 있습니다.

### 슬라이드 17. 장소 제안 — 25초

목록에 없는 장소나 달라진 정보는 Google Form으로 제안할 수 있습니다. 제출 내용은 바로 공개되지 않습니다. 관리자가 위치와 입력값을 확인하고 승인한 내용만 최종 데이터에 반영합니다.

## 03 개발·협업

### 슬라이드 18. 바이브코딩 — 25초

개발에는 Codex를 바이브코딩 도구로 활용했습니다. 팀이 문제, 요구사항과 판단 기준을 정하고, Codex는 기존 코드 분석, 기능 구현, 오류 원인 분석과 문서화를 지원했습니다. 최종 선택과 검수는 사람이 담당했습니다.

### 슬라이드 19. 협업 개발 과정 — 30초

화면의 사진처럼 먼저 팀에서 사용자 흐름과 요구사항을 함께 정했습니다. 이를 자연어 작업 지시로 정리해 구현하고, 모든 팀원이 실제로 사용하며 디자인, 데이터와 예외 상황을 확인했습니다. 확인한 결과를 다음 개선사항에 반영하는 방식으로 협업했습니다.

### 슬라이드 20. 팀 역할 — 40초

양다경은 팀장으로서 서비스 기획, 전체 개발, 데이터 구조, 기능 통합과 배포를 담당했습니다. 전아진은 사용자 관점의 UI·UX와 발표 자료 시각화를 주도했습니다. 양채영은 초기 데이터 정제, 누락값 확인, 장소 제안·수정 Google Form을 담당했습니다. 역할은 나눴지만 기능, 데이터와 사용성 검증은 세 명 모두 함께했습니다.

### 슬라이드 21. 시스템 구조 — 35초

앱은 Streamlit으로 만들었습니다. 메인 장소는 jeju-irang CSV에서 읽고, 즐겨찾기는 비공개 Google Sheets의 bookmarks 탭에 저장합니다. 장소 제안은 Google Form과 Apps Script를 거쳐 검수되며, 외부 API는 사용자 검색 순간이 아니라 관리와 데이터 보강 과정에서 사용합니다.

## 04 데이터

### 슬라이드 22. 기본 데이터 — 30초

기본 장소와 가족 편의정보는 한국문화정보원의 전국 가족 유아 동반 가능 문화시설 위치 데이터를 활용했습니다. 여기서 제주 지역을 선별하고 형식을 정리했습니다. 원본 기준 파일명은 20221031이고 업데이트가 수시 1회성이므로 모든 정보가 실시간 최신이라고 표현하지 않았습니다.

### 슬라이드 23. API 보강 — 30초

지역과 위치는 VWorld 검색 API 2.0으로 확인했습니다. 장소 설명, 연락처와 운영정보는 한국관광공사 국문 관광정보 서비스의 KorService2로 보강했고, 대표 이미지는 관광사진 정보 API를 활용했습니다.

### 슬라이드 24. 검수 흐름 — 35초

사용자 제안이 들어오면 Apps Script가 응답을 정리하고 VWorld 후보를 찾습니다. 관리자가 후보, 주소와 입력값을 확인한 뒤 승인하면 최종 CSV가 생성됩니다. 앱 실행 중에는 API를 직접 호출하지 않기 때문에 API가 일시 중단돼도 기본 검색은 계속 사용할 수 있습니다.

### 슬라이드 25. 데이터 규모 — 25초

현재 장소 데이터는 260곳, 27개 컬럼입니다. 시설유형은 전시·기념관 133곳, 관광지 84곳, 영화·연극·공연 43곳입니다. 공간유형은 실내 145곳, 실외 58곳, 실내와 실외를 함께 이용하는 장소가 57곳입니다.

### 슬라이드 26. 장소 스키마 ① — 25초

첫 번째 그룹은 장소를 식별하고 검색과 지도 표시에 쓰는 정보입니다. 고유 ID와 장소명, 시설과 공간유형, 행정시와 읍면동, 검색 권역, 도로명 주소와 위경도로 구성했습니다.

### 슬라이드 27. 장소 스키마 ② — 25초

두 번째 그룹은 방문 전에 확인해야 하는 운영정보입니다. 연락처, 운영시간, 휴무일과 주차정보가 있고, 입장료와 연령제한은 여부와 상세 내용을 나누어 저장했습니다.

### 슬라이드 28. 장소 스키마 ③ — 25초

세 번째 그룹은 서비스의 핵심인 가족 편의정보입니다. 수유실, 유모차 대여, 기저귀 교환대와 도민 할인을 포함합니다. 공식 홈페이지, 예약 링크, 사진, 소개와 방문 참고사항도 같은 장소 ID에 연결했습니다.

### 슬라이드 29. 즐겨찾기 데이터 — 25초

즐겨찾기는 장소 데이터와 분리했습니다. 즐겨찾기 ID와 닉네임, 장소 연결 키, 저장 시각, 비밀번호 salt와 해시, 메모와 사용자 카테고리를 저장합니다. 하나의 장소를 여러 사용자가 저장할 수 있어 place ID를 기준으로 1대 다 관계입니다.

### 슬라이드 30. 추천 규칙 — 25초

상세 화면의 비슷한 장소 추천은 rules CSV로 관리합니다. 비교할 컬럼, 비교 방식, 가중치, 기준값과 사용 여부를 데이터로 바꿀 수 있습니다. 규칙 파일이 없거나 잘못돼도 앱의 기본 규칙으로 계속 동작합니다.

## 05 데모

### 슬라이드 31. 데모 ① 실내 나들이 — 45초

첫 번째는 제주 거주 보호자의 상황입니다. 비 오는 주말에 아이와 갈 곳을 찾는다고 가정하겠습니다. 실내, 주차 가능, 기저귀 교환대 조건을 선택하고 결과를 확인하겠습니다. 그중 한 장소를 열어 운영정보와 편의시설이 같은 화면에 나오는 것을 보여드리겠습니다.

시연 순서: `장소 찾기 → 실내 선택 → 주차 가능 → 기저귀 교환대 있음 → 결과 카드 → 상세 화면`

### 슬라이드 32. 데모 ② 가까운 장소 — 45초

두 번째는 아이 동반 여행객의 상황입니다. 위치 사용을 허용한 뒤 갤러리나 표에서 가까운 순으로 후보를 좁히겠습니다. 그중 한 장소를 선택하고, 지도에서는 정렬이 아니라 실제 위치와 이동 동선을 확인하겠습니다.

시연 순서: `위치 사용 → 갤러리·표에서 가까운 순 → 후보 선택 → 지도에서 위치 확인 → 상세 화면`

### 슬라이드 33. 데모 ③ 나들이 목록 — 45초

세 번째는 반복 검색을 줄이는 시나리오입니다. 닉네임과 비밀번호로 시작하고 마음에 든 장소를 즐겨찾기에 저장하겠습니다. 카테고리를 만들고 메모를 남긴 뒤 나만의 목록에서 다시 확인합니다.

시연 순서: `닉네임·비밀번호 → 즐겨찾기 → 카테고리 → 메모 → 저장 목록`

## 06 문제 해결

### 슬라이드 34. 부족한 데이터 보강 — 45초

기본 공공데이터만으로는 일부 장소의 정확한 지역과 위치, 관광 설명과 대표 사진이 부족했습니다. 지역과 좌표는 VWorld 검색 API 2.0으로 확인하고, 관광정보는 TourAPI KorService2, 대표 이미지는 관광사진 API로 보강했습니다. 다만 비슷한 장소명이나 오래된 정보가 연결될 수 있어 API 결과를 자동으로 공개하지 않았습니다. 장소명, 주소와 좌표를 관리자가 확인한 뒤 승인된 정보만 최종 CSV에 반영했습니다.

### 슬라이드 35. 지속되는 즐겨찾기 — 45초

로컬에서는 bookmarks CSV에 저장할 수 있지만, 배포 서버의 파일은 재시작이나 재배포 뒤에도 유지되는 공용 저장소가 아닙니다. 여러 사용자가 함께 쓰는 서비스에서 즐겨찾기가 사라질 수 있는 문제였습니다. 그래서 배포 환경의 즐겨찾기를 비공개 Google Sheets의 bookmarks 탭으로 분리했습니다. 장소와는 place ID로 연결하고, 새로고침 뒤에도 목록이 남는지 확인했습니다. 저장소 연결에 문제가 생겨도 장소 검색과 상세보기까지 함께 멈추지 않도록 오류를 분리해 안내합니다.

### 슬라이드 36. 개인정보 보호 — 40초

회원가입을 없애면 사용은 간편하지만 즐겨찾기를 안전하게 구분해야 합니다. 닉네임과 비밀번호를 사용하되 비밀번호 원문은 저장하지 않고, 사용자별 salt와 PBKDF2 해시만 저장했습니다. 현재 위치도 세션의 거리 계산에만 사용하고 CSV나 Google Sheets에는 남기지 않습니다.

## 07 한계·계획

### 슬라이드 37. 현재 한계 — 40초

현재는 이용 행동 데이터가 없어 인기순 기능이 없습니다. 260곳 중 사진은 83곳, 설명은 22곳, 예약 링크는 3곳에만 있어 데이터 보강이 필요합니다. 일부 기본 데이터는 2022년 기준입니다. 또한 관리자가 검수하면 최종 CSV는 생성되지만, 이 파일을 프로젝트의 data 폴더에 있는 jeju-irang CSV로 교체하고 반영해 Streamlit을 다시 배포하는 마지막 단계는 아직 수동입니다.

### 슬라이드 38. 다음 단계 — 35초

다음에는 TourAPI와 관리자 검수로 사진, 설명과 편의정보를 보강하겠습니다. 제주특별자치도관광협회의 관광지 입장요금표로 요금정보도 강화하겠습니다. 검수된 CSV의 반영과 배포를 자동화하고, 이후 이용 데이터를 기반으로 인기순을 제공한 뒤 표준화한 구조를 전국으로 확대할 계획입니다.

## 마무리

### 슬라이드 39. 더 간단한 외출 결정 — 20초

제주아이랑은 흩어진 아이 동반 장소 정보를 한곳에 모아 우리 가족의 조건으로 검색하고 비교하고 저장할 수 있게 만들었습니다. 화면의 QR 코드로 지금 이용해 보실 수 있습니다. 제주에서 시작한 이 서비스가 가족의 외출 결정을 더 간단하게 만드는 것이 저희의 목표입니다. 감사합니다.
